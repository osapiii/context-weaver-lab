# メンテナンス・更新ガイド（デッキ構築モードの「既存更新」サブフロー）

> 🧭 **このガイドは「デッキ構築モード → 既存デッキ更新」を担当します。**
> ユーザーが作った/これから作る **個別デッキ** の修正依頼がここに来ます。
>
> **スキル自体のデザイン資産を変える依頼** — たとえば brand カラー変更、
> テンプレ実装のコード改変、新テンプレ追加、ロゴ差し替え等は **デザイン更新モード**
> の管轄なので `references/alt-modes/design-update-mode.md` を読むこと。混同しないこと。
>
> 判断に迷ったら SKILL.md の「モード分岐」→「判断に迷ったら」表を確認。
> 核は「その変更はこのデッキだけの話か、全デッキに波及するか」。
> 個別デッキだけなら本ガイド、全デッキ波及ならデザイン更新モード。

---

## いつこのガイドを使うか

ユーザーから以下のようなメッセージが来た時：

- **テンプレ ID を含む修正依頼**: `VISUAL-1-profile の写真領域を小さくしたい`
- **既存デッキの特定箇所への依頼**: `S5 の比較カラムを 3 カラムに変えて`
- **「直して」「調整して」「ブラッシュアップして」** 系の指示
- **既存ファイルの添付 + 変更要望**

新規デッキ依頼（「資料を作って」）は `_common/workflow.md` のフロー、**既存物の改善依頼はこの maintenance-guide のフロー** を使います。

---

## テンプレ参照 ID の仕組み

各ページテンプレートには、安定した **参照 ID** が割り当てられています：

| ID 形式 | 例 | 意味 |
|---------|-----|------|
| `{TEMPLATE-ID}-slug` | `VISUAL-1-profile` | テンプレ ID + slug |
| `LIST-1-content` | 標準コンテンツ |
| `DIAGRAM-1-matrix-2x2` | マトリクス 2×2 |

ユーザーは `_team-reference/components/index.html` や個別テンプレ HTML の **コピーボタン** でこの ID をクリップボードにコピーし、Claude との会話に貼り付けます。

Claude は ID を受け取ったら、以下の関連ファイルを特定します：

| ID が指す実体 | 該当ファイル |
|-------------|---------|
| HTML リファレンス | `_team-reference/components/NN-slug.html` |
| PptxGenJS 実装 | `scripts/example-deck.js` の対応ブロック（コメント `// NN. タイトル` で検索） |
| デザイン仕様 | `references/_common/design-system.md` の対応エントリ |
| テンプレ選定ガイド | `references/_common/slide-patterns.md` |

## 2 段階の判断 — 小変更は即、大変更は指示書

更新依頼を受けたら、まず **変更の規模を判定** します。

### 🟢 小変更 — 即適用していい

**判定基準**: 以下のすべてに当てはまる
- 単一スライド・単一要素の変更
- 視覚的な影響が局所的（他要素との関係性が崩れない）
- 意味の変更を伴わない
- 共通ヘルパー・トークンに触れない

**具体例**:
- 文言変更（タイトル・サブコピー・箇条書きの文字）
- 数値の置き換え（金額・日付・統計値の差し替え）
- 色トーン切替（`gray` → `purple`、`gray` → `amber`）
- 1 要素の座標微調整（既存要素の位置を数値で微動）
- 要素の単純追加（タイル 1 枚増やす、ブレット 1 項目追加）
- 誤字修正・typo 訂正
- 固有名詞の置換（「A 社」→「B 社」）

**実行手順**:
1. 該当箇所を view で確認
2. str_replace で直接変更
3. PPTX 生成 → 目視確認
4. 完了報告（「S15 のサブコピーを更新しました」）

### 🟡 大変更 — 必ず指示書フロー

**判定基準**: 以下のいずれかに当てはまる
- 複数要素の再配置・サイズ変更
- 情報構造の変更（何を見せるかの変更）
- 他テンプレや共通ヘルパーに影響する
- ブランドトークン自体を変える
- 「全体を見直したい」「ブラッシュアップして」等の曖昧な指示

**具体例**:
- テンプレ番号の変更（17 Tile 2×2 → 18 Tile 3×2）
- レイアウト全面見直し（要素の再配置・サイズの複数変更）
- 複数の要素を同時に動かす
- 共通ヘルパー（`addTitleBlock`, `addChromeNav` 等）の変更
- `tokens.js` / `themes.js` のブランド値編集
- 新しい表現パターンの追加（`design-system.md` の更新を伴う）
- カラーテーマの新規追加
- 複数スライドに跨る一貫した変更（「全スライドの余白を詰めて」等）

**実行手順**: 次節「大変更の指示書フロー」を参照。

### 迷ったら？

**指示書を出す側に倒す**。小変更として処理してみて、結果が不確実に感じたら、あらかじめ指示書で合意を取る。

---

## 大変更の指示書フロー（4 ステップ）

### Step 1 — ユーザーの依頼内容を確認

受け取る情報：
- **対象 ID**（例: `VISUAL-1-profile`）または **既存デッキ中の該当スライド指定**
- **変更したい内容**（「〜したい」「〜が気になる」等）

情報が不足していれば、この段階で質問：
- 「どのテンプレ／スライドですか?」（ID 未指定の場合）
- 「どう変えたいですか?」（要望が曖昧な場合）
- 「既存の他のスライドとの整合性、どう扱いますか?」

### Step 2 — 関連箇所の特定と影響調査

ID を受けたら、以下を順に実行：

1. **実装コードを読む**: `scripts/example-deck.js` の該当ブロックを view
2. **HTML リファレンスを読む**: 必要なら `_team-reference/components/NN-slug.html`
3. **デザイン仕様を確認**: `references/_common/design-system.md` で「このテンプレが使っている表現パターン」を把握
4. **影響範囲を調査**: 同じ表現パターンを使っている他のテンプレは? 共通ヘルパーに手を入れる必要があるか?

### Step 3 — 更新指示書 (Markdown) を生成

以下の構造で **Markdown ファイル** として `logs/...` 配下に出力:

```markdown
# {対象} 更新指示書 v1

## 1. 対象
- **ID**: VISUAL-1-profile
- **ファイル**: scripts/example-deck.js の `// 15. プロフィール` ブロック
- **関連**: _team-reference/components/15-profile.html

## 2. 変更内容
{ユーザーの依頼を明確な変更指示に言語化}
- 例: 顔写真領域を現在の 2.0"×2.5" から 1.6"×2.0" に縮小
- 例: 右側カードの幅を拡大して 3 枚並びを維持

## 3. 変更理由
{なぜこの変更が妥当か}
- 例: 強みカード 3 枚の情報量が本質、顔写真は副次的という優先順位

## 4. 影響範囲
{他のテンプレ・ヘルパー・トークンに影響するか}
- 今回の変更はスライド 15 のローカル変更のみ、他テンプレ・共通ヘルパーに影響なし
- または: 共通ヘルパー addProfileHeader を変更するため、他テンプレへの影響確認必要

## 5. 触らない箇所
{このチケットで編集してはいけない箇所を明示}
- タイトルブロック（addTitleBlock）
- クローム（addChromeWithNav）
- 左端ストライプ、ロゴ配置、フッター
- 他のテンプレ（01〜14, 16〜25）

## 6. 変更前後の差分（概要）
{必要ならコードの新旧対比を疑似コードで}
- Before: `w: 2.0, h: 2.5`
- After:  `w: 1.6, h: 2.0`

## 7. 確認のお願い
この指示書で変更を進めてよろしいですか?
```

### Step 4 — 承認後にコード変更

承認（「OK」「進めて」等）を受けたら：

1. `str_replace` で該当箇所のみを編集（全文書き換えない）
2. `scripts/example-deck.js` を実行して PPTX 生成
3. PDF → 画像化で目視確認
4. 問題なければ完了報告、必要なら微調整

**明示的な承認がない限り変更には進まない**。曖昧な返事は承認と見なさない。

## 触ってはいけない箇所（全テンプレ共通）

小変更・大変更どちらでも、**特別な指示がない限り触らない** 箇所：

### 1. デザイントークン値 (`assets/tokens.js`, `assets/themes.js`)

- 色、サイズ、レイアウト値、フォント名
- 変更する場合は **ブランドガイドラインの更新** として別チケット（必ず大変更フロー）

### 2. 共通ヘルパー関数

- `addTitleBlock`, `addChrome*`, `setCanvasBg`, `addChromeNav`
- これらを変更すると全スライドに影響するため、専用の指示書で合意してから

### 3. クローム要素

- 左端 purpleSoft ストライプ
- 上部ナビチップ
- 右上ロゴ、フッター、ページ番号
- 上下の 2 色バー（表紙・閉じ）

### 4. 他のテンプレのコード

- 依頼対象以外のスライドブロックは読むだけ、編集しない

## 変更後の自己チェック

実装後、以下を確認してからユーザーに報告：

- [ ] 対象スライドだけが変更されているか（他スライドへの意図しない影響なし）
- [ ] タイトルブロック・クロームが変わっていないか
- [ ] 新しいハードコード色が入っていないか（`C.brand` 等トークン経由になっているか）
- [ ] Noto Sans JP (`F.jp`) が指定されているか
- [ ] PPTX 生成が成功するか
- [ ] 生成した画像で崩れがないか

## 更新時の典型的な失敗と対処

### 失敗 1 — 小変更判定で大変更を強行する

対処: 変更中に「あ、これは他にも影響する」と気づいたら、いったん止めて指示書フローに切り替える。「思ったより影響範囲が広いので、指示書を出させてください」とユーザーに伝える。

### 失敗 2 — 他のテンプレまで変えてしまう

対処: `str_replace` の前に該当ブロックを view し、対象以外のコードを触らないことを確認。全文書き換えは禁止。

### 失敗 3 — 共通ヘルパーを改変してしまう

対処: 共通ヘルパー編集は必ず大変更フロー。依頼時にユーザーに「これは共通部分なので全スライドに影響しますが、本当にこれで良いですか?」と確認する。

### 失敗 4 — ハードコード値を入れる

対処: 色やサイズは必ず `C.brand` や `L.marginX` 等のトークン経由で。リテラル値を書きたくなったら `tokens.js` への追加を提案する。

### 失敗 5 — 大変更で承認前に変更を走らせる

対処: 更新指示書を出したら **明示的な OK** を待つ。「確認してみます」「そうですね」等の曖昧な返事で進めない。

> - 「整える」の解釈が Claude 任せになる
> - 触った箇所がユーザーに見えない

## HTML リファレンスの扱い（重要）

`_team-reference/components/*.html` は **チームのビジュアル参照用** で、Claude の実行には関わりません。そのため：

- **PPTX 実装 (`scripts/example-deck.js`) がソース・オブ・トゥルース**
- HTML リファレンスは PPTX の変更に対して **多少ズレていても構わない**
- 「どの程度正確か」が気になる場合は `_team-reference/components/preview/slide-NN.jpg` を参照（こちらは PPTX から自動生成される正確な画像）

### HTML リファレンスを更新すべきケース

以下の時だけ HTML も同期更新する：
- デザイン協議で「こう見える」を共有する必要がある
- 新しいテンプレを追加した時（新テンプレの HTML が必要）
- 大きなブランド変更（色体系の刷新など）

**小さな変更（文字サイズ微調整、余白変更）では HTML 側は更新しなくて良い**。PPTX の `preview/slide-NN.jpg` が常に最新なので、そちらで確認すれば十分。

### プレビュー画像の再生成

PPTX を変更したら、プレビュー画像を再生成する簡易手順：

```bash
cd scripts
node example-deck.js
python3 /mnt/skills/public/pptx/scripts/office/soffice.py --headless --convert-to pdf enostech-example.pptx
pdftoppm -jpeg -r 110 enostech-example.pdf slide
cp slide-*.jpg ../_team-reference/components/preview/
```

## まとめ

- **変更の規模で判定**: 小変更 🟢 は即適用、大変更 🟡 は指示書フロー
- **迷ったら指示書側に倒す**
- **大変更は明示的な承認を待つ** — 曖昧な返事で進めない
- **HTML リファレンスは多少ズレても OK、PPTX が正解**

このフローを守ることで、チーム全員が安心してデザインメンテを依頼できる状態を維持できます。
