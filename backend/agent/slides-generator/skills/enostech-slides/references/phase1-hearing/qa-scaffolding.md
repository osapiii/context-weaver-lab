# Phase 1 — QA 駆動モードのヒアリング scaffolding

> 「読者が事前に持っている疑問・懸念のリスト」を中心にヒアリングを組み立てる。
>
> 適用条件: ユーザーが明示的に「QA 駆動で作って」「疑問リストから組みたい」「Q&A 形式で」
> と発話した時、または Phase 1 後半で Claude が読者像から「QA 駆動が筋が良い」と判断した時。

---

## 0. なぜ QA 駆動モードか

普通のヒアリングは「何をデッキに入れたいか」をユーザーに書き出してもらう設計者目線の作業。
これは設計者にとっては自然でも、**普通のユーザーには難しい**。普通の人がデッキを欲しがる
時に頭にあるのは設計図ではなく **「こういうことが知りたい / ここが不安」という疑問のリスト**。

QA 駆動モードはこの自然な発想に寄り添う:

- ユーザー視点: 「自分の疑問・懸念」を書き出すだけで OK
- AI 視点: 各疑問に答えるための材料を組み立てればよい
- デッキ視点: 「N 個の疑問が解消される」が完了条件 (StructureQA で機械検証)

---

## 1. Claude の進め方 — 5 ステップ

### Step 1. テーマ・読者の確認 (通常通り)

R1-1 (Before→After) と R1-2 (こだわり / NG) は QA 駆動でも変わらない。
読者像とテーマを 2 文程度で固める。

### Step 2. 叩き台の Q を 5-8 個提示する

**ここが QA 駆動モードの肝**。ユーザーに「どんな疑問がありますか?」と白紙で聞かない。
Claude が読者像 + テーマから推測した叩き台を **5-8 個提示する**。ユーザーは追加 /
削除 / 書き換えで反応すれば良い。

#### 叩き台 Q の作り方

質問の型 (kind enum 6 種) を念頭に **配分良く** 並べる。
ある型に偏った 5 個より、多様な 5 個の方が読者の頭に近づく。

| 型 | 意味 | 例 |
|----|------|-----|
| `definitional` | 「○○とは何か」(概念導入) | 「ベアフットシューズって何?」 |
| `comparative` | 「A と B の違いは」(選択判断) | 「ミニマリスト系とトラディショナル系の違い?」 |
| `decisional` | 「○○すべきか」(行動判断) | 「自分の足には合うのか?」 |
| `how_to` | 「○○のやり方は」(手順) | 「最初の 1 足はどう選ぶ?」 |
| `risk` | 「○○の懸念は」(注意点) | 「いきなり履いて怪我しない?」 |
| `other` | その他 | (上記に当てはまらない時のみ) |

**配分目安 (5-8 件の場合):**

| 件数 | definitional | comparative | decisional | how_to | risk | other |
|------|--------------|-------------|------------|--------|------|-------|
| 5 件 | 1 | 1 | 1 | 1 | 1 | 0 |
| 6 件 | 1 | 1 | 1 | 2 | 1 | 0 |
| 7 件 | 1-2 | 1 | 1 | 2 | 1-2 | 0 |
| 8 件 | 2 | 1 | 1 | 2-3 | 1-2 | 0 |

これは目安で、テーマによって偏りがあって OK (例: トラブルシュートデッキは
risk と how_to が多い、概念導入デッキは definitional が多い)。

#### Q の text の書き方 (10-80 字)

- 読者が **実際に発する** 言葉で書く: 「Vf の決め方は?」より「LED に何 Ω つければいい?」
- 一文一疑問: 「A と B の違いは何で、どっちを選ぶべき?」を 2 件に割る
- 専門用語は読者レベルに合わせる: 初級者向けなら専門用語は使わない
- 80 字超は焦点ボヤけ → 分割推奨

### Step 3. 各 Q に provisionalDirection (暫定回答方向性) を添える

**Phase 1 段階の目的: ユーザーが「方向性が合っているか」を判断できる材料を渡す**。
完璧な答えは Phase 2 で精緻化されるので、ここでは 1-2 文で「こういう方向で答える予定」
を示す。

#### provisionalDirection の書き方

- **AI 内部知識ベースで先出し**して OK (Phase 1 の体感を軽くするため)。
  実裏取りは Phase 2 入口で `web_search` を使って精緻化する。
- 1-2 文 / 30-100 字程度
- 結論先行: 「○○は××です。理由は△△」の順で書く
- 「分かりません」は使わない。曖昧でも方向性は示す

**良い例:**
```
Q1: 電子工作の抵抗ってどう選べばいいの?
provisionalDirection: 「LED を点灯させる用途なら、電源電圧と LED の順電圧の差を
電流で割って抵抗値を出します。一般用途なら 220Ω で十分なケースが多いです。」
```

**悪い例:**
```
Q1: 電子工作の抵抗ってどう選べばいいの?
provisionalDirection: 「色々な選び方があります」  ← 方向性が無い
provisionalDirection: 「Vf=2V / If=20mA の典型値で R = (5-2)/0.02 = 150Ω となり、
              安全マージンを取って 220Ω E24 系列を選択します」  ← Phase 1 段階としては詳細すぎ
```

### Step 4. ユーザーに提示 → 反応を受ける

提示フォーマット:

```
このデッキで解消したい疑問・懸念を {N} 個に整理しました。
ご確認ください。追加・削除・書き換えがあれば指示してください。

Q1 [how_to] 電子工作の抵抗ってどう選べばいいの?
   → LED を点灯させる用途なら、電源電圧と LED の順電圧の差を電流で割って
     抵抗値を出します。一般用途なら 220Ω で十分なケースが多いです。

Q2 [definitional] カラーコードって何?
   → 抵抗器の本体に印刷された 4-5 本の色帯で、抵抗値と誤差を表します。

Q3 [comparative] 直列と並列で抵抗値はどう変わる?
   → 直列は和、並列は逆数の和の逆数で、用途で使い分けます。

(...)

これで方向性 OK でしたら Phase 2 に進みます。
他に気になる点・調べたい点があれば教えてください。
```

ユーザー反応のパターン:
- **「OK 進めて」** → そのまま Phase 2 へ
- **「Q3 を削って、代わりに ○○ について追加」** → Q list を編集して再提示
- **「Q1 はもっと初心者向けに」** → text と provisionalDirection を書き直して再提示
- **「全然違う、こういう疑問が知りたい」** → 大幅再構築

### Step 5. 承認後 plan.json の準備に進む

Phase 1 の出口 = 確定した questions[]。ただし Phase 1 では:
- ✓ id / text / kind / provisionalDirection を埋める
- ✗ shortSummary / sectionIndex は **空のまま**で良い (Phase 2 で確定)
- ✗ doc.phase2_locked は **false のまま** (Phase 2 で true に切替)

Phase 2 入口で `web_search` を回して provisionalDirection を精緻化、章割りが固まった
段階で sectionIndex を埋め、各 Q に shortSummary (≤30 字) を絞る。最終的に Phase 2
完了時に doc.phase2_locked: true をセットして StructQA-52 を起動 → 全 Q の
shortSummary / sectionIndex を機械検証。

## 2. 件数の目安

| 件数 | 性質 | 該当例 |
|------|------|--------|
| 2 件 | 1 つのテーマで 2 軸の疑問 | 「○○とは何か」「自分はどうすべきか」のような根源 2 問 |
| 3-5 件 | デッキの典型 | 標準的な学習・提案デッキ |
| 6-8 件 | 多面的な解説 | 1 ジャンル全景・FactHub 機能ガイド等 |
| 9-12 件 | 大きめのリファレンス系 | 全機能カタログ・年次レビュー |
| 13-15 件 | 重量級リファレンス | 上限。文字 9pt + 行高 0.21" でレイアウト維持 (StructQA-50 の上限) |
| 16 件以上 | StructQA-50 fatal | 2 デッキ分割を促す |

## 3. NG パターン (やってはいけない)

### 3.1 Claude が叩き台を出さずユーザーに白紙で書かせる

「どんな疑問・懸念がありますか?」とだけ聞いて沈黙するパターン。これは元の課題
(ヒアリング入力が大変) を解消できない。**叩き台を必ず出す**。

### 3.2 Q を全部 how_to に偏らせる

QA 駆動モードを「FAQ 形式」と勘違いして全部「○○のやり方は」になるパターン。
読者の頭の中はもっと多様。kind enum を埋めるくらいの気持ちで。

### 3.3 provisionalDirection を空にする / 「Phase 2 で書きます」と濁す

ユーザーが Phase 1 で承認できる材料が無くなり、結局 Phase 2 まで判断保留 →
手戻りが大きくなる。**1-2 文でも方向性は示す**。AI 内部知識ベースでも OK。

### 3.4 shortSummary を Phase 1 で確定させようとする

shortSummary は **30 字** に絞る最終回答。Phase 1 段階では
provisionalDirection の方向性が合っているかが論点で、30 字に絞り込む段階ではない。
**Phase 2 完了時に確定** が正しい運用。

## 5. opt-in の判断基準

Phase 1 で QA 駆動モードを採用するか迷ったら:

**採用すべき (true)**:
- 読者が事前に明確な疑問を持って読む (学習デッキの大半 / 提案デッキ)
- 「○○について解説」より「○○のここが知りたい」と読者が言いそうなテーマ
- ユーザーから「QA 形式で」「疑問リストから」と発話があった

**採用しなくて良い (false / default)**:
- 出来事報告 — 読者が事前に Q を持っていない
- 事例カタログ — 比較軸ベースで読まれる
- 章構造で十分整理できるテーマ
- ユーザーから明示の opt-in 指示が無い

迷ったら **false** で進めるのが安全。後から qa_driven=true に切り替えるのは比較的容易だが、
Q の枠で組んでしまったストーリーを Q なしに戻すのは手戻り大。
