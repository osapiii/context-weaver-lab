# バイパスモード — ユーザー確認ステップだけ省略する薄いオーバーレイ

> **重要原則**: bypass モードは独自フローではない。
> **通常の `workflow.md` を全工程そのまま実行**し、
> **「ユーザー承認待ち」ステップだけを省略**するモード。
>
> 処理内容（HTML 指示書の生成、M? VIOLATION チェック、PNG 化、QA、decks/ 昇格 等）は
> 通常モードと完全に同じ。**スキップするのは「待つこと」だけ**。

---

## なぜこの原則を守るか

bypass モードが Phase ごとに独自定義を持つと、「Phase 2 では HTML 指示書を出さない」のような
**通常フローからの乖離** が混入し、M? VIOLATION チェックが効かない・トレーサビリティが
失われる等の品質ブレが発生する。

そのため bypass-mode.md には **どのファイルでどの処理をするか** を書かない。
**どの確認ステップを省略するか** だけを書く。

---

## 中間ファイルパスは `<workdir>` 変数化

このドキュメント中の `<workdir>` は中間ファイル保存先を指す **変数**:

| シーン | `<workdir>` の値 |
|---|---|
| 単発バイパス（このドキュメント標準）| `/tmp/bypass-log/`（デフォルト）|
| 任意のスクラッチ運用 | ユーザーが指定した任意のディレクトリ |

スクリプト群はすべて出力ディレクトリを引数で受け取る作り。
`<workdir>` は「どこに置くか」の慣習で、bypass モード時に親エージェントが選んで一貫させる責務。

## 🔴 バイパスモード絶対ルール

### R-BYPASS-1 プロンプトに `--bypass` が含まれていたら必ず本ファイルを読む

スキル発動時に最初にチェックする。含まれている場合、**確認待ちステップをスキップして**
Phase 1 → Phase 4 まで完走する。ただし必須 4 項目（読者・目的・分量・テーマ）が
揃っていない時のみ 1 回だけまとめて質問する。判定は大文字小文字を区別しない
（`--BYPASS` / `--Bypass` でも発動）。

モード B（スキル本体のデザイン更新）には bypass を適用しない。

### R-BYPASS-2 バイパスモードの Phase 4 で自動修正リトライは最大 2 回まで

Phase 4 でユーザー判断ステップが省略される代わりに、Claude が自動修正を試みる。
最大 2 回までリトライし、それでも解消しない問題 or 3 件以上の問題が最初から
検出された場合は、進行を止めてユーザーに報告する。残存問題は完了レポートに必ず記載する。

### R-BYPASS-3 独自の Phase 定義を絶対に書かない

bypass-mode.md には「Phase X ではこうする」という処理内容を書かない。
書いてよいのは以下 3 点のみ:

1. どの**ユーザー確認待ちステップ**をスキップするか（後述の「省略する確認ステップ一覧」）
2. bypass 固有のフォールバック（Phase 4 自動修正リトライ等）
3. 必須情報チェック・完了レポート規約

処理内容は通常フロー側（workflow.md / 各 phase README）が **唯一の正**。
両者で同じことを書くと二重メンテになり、片側だけ更新されて品質がブレる。

### R-BYPASS-5 QA は scripts/run-qa.py を必ず踏む

bypass モードでも、Phase 2 完了直前と Phase 4 で **`scripts/run-qa.py`** を実行する。
これは「Markdown チェックリストを目で読んだだけで OK と判定する」事故を構造的に防ぐため。

**実行手順**:
```bash
# Phase 2 完了直前 (機械検証 + 構成・引用の手動チェック雛形書き出し)
python3 scripts/run-qa.py phase2 --plan decks/{slug}/plan.json --bypass
# → 機械検証で fatal なら exit 2 で進行停止
# → pass なら手動 QA セルフレポート (qa-self-report-phase2.md) を雛形書き出し → exit 3
# → 雛形を編集 (各ルールに status と note を埋める)
# → 同じコマンドに --apply-manual を付けて再実行 → plan.json に反映

# Phase 4 (ビジュアル検査)
python3 scripts/run-qa.py phase4 --plan decks/{slug}/plan.json
# → pptx-to-images.sh で PNG + コンタクトシート生成
# → VQA セルフレポート (qa-self-report-phase4.md) を雛形書き出し → exit 3
# → 雛形を編集
# → --apply-manual で再実行 → plan.json に反映
```

**Exit code の意味**:
- `0`: QA pass、次フェーズに進める
- `2`: 機械検証で fatal 違反、修正必須
- `3`: 手動 QA セルフレポートが未記入、編集必須
- `4`: 下流スクリプトの異常 (build エラー等)

**省略してはいけない理由**: bypass モードで急ぐと「コンタクトシート目視 OK」の
1 行で済ませがちだが、12 観点を 1 つずつ「pass/fail と理由」を書き出すと、
普段見落としていた視認性問題やレイヤー重なりに必ず気付ける。1 件あたり 30 秒、
全 12 件で 6 分。この 6 分を惜しまない。

**fail があった時**: そのままユーザーに報告 (進行停止) または bypass の自動修正
リトライ枠 (R-BYPASS-2) で対応する。雛形に書いた `note` がそのまま完了レポートに
転記されるので、後から読み返せる。

### R-BYPASS-4 裏取り (web_search) は bypass モードでも省略不可

ユーザーから受け取った数値・固有名詞・第三者事例 (調査機関名、企業名、業界トレンド) は、
bypass モードでも **その場で web_search を実行して裏取り** する。

**実行しなかった場合の挙動**:
1. 該当主張の `ref_table` を空にせず、`title: "[要裏取り] {ユーザー発言の引用}"` 形式で
   `doc.references[]` に「裏取り未実施」エントリを残す
2. 完了レポートで「裏取り未実施 N 件」を必ず列挙

**判定基準** (これらが含まれていたら裏取り対象):
- 数値: `\d+%`, `\d+倍`, `\d+件`, `20XX 年` 等の絶対値・割合・年次
- 第三者名: IPA / Gartner / IDC / McKinsey / 総務省 / 経済産業省 等の調査機関
- 業界トレンド主張: 「市場規模が〜」「導入企業の〜割が」等

**自動検証**: `python3 scripts/run-qa.py phase2 --plan decks/{slug}/plan.json --bypass` で
`RefQA-02` が fatal 化される。

> なぜ bypass でも省略不可か: ファクトベースは ENOSTECH デッキの読者信頼性の根幹。
> 「Claude が裏取りせずに数値を載せた」が一度でもあると、デッキ全体の信頼が崩れる。
> 5 秒の web_search を惜しんで億単位の信頼を失わない。

## トリガー

プロンプト内に `--bypass` という文字列があれば発動する。モード A（デッキ構築）にだけ適用。

```
/enostech-slides --bypass

[以降に依頼内容]
```

`--bypass` が含まれていなければ、通常通り全フェーズフローを実行する。
モード B（スキル本体のデザイン更新）には bypass を適用しない。デザイン変更は
全デッキに波及するため、ビジュアルチェックと承認を省略してはいけない。

---

## 必須情報（揃っていなければ 1 回だけ質問）

bypass モードでも、最低限の情報が無いと望まぬデッキができる。以下 4 項目のうち
揃っていないものがあれば、**1 回だけ**まとめて確認する：

| # | 項目 | 例 |
|---|------|-----|
| 1 | **読者** | 「既存ユーザーのデータアナリスト」 |
| 2 | **目的** | 「v2.0 の 3 つの新機能の理解」 |
| 3 | **分量** | 「15 枚前後」 |
| 4 | **テーマ** | `enostech` / `corporate` / `nature` / `warm` / `mono` |

**省略可能**: サブコピーのトーン、こだわりの表現、参考素材、図解のテイスト
など（デフォルトで埋める。図解パターンは `_common/diagram-patterns.md` の DIAG-XX
で canonical に定義されているので bypass モードでも聞かない）。

プロンプト内でこれら 4 項目が埋まっていれば、確認なしで即実行する。

## 省略する確認ステップ一覧（bypass の本体）

下表が bypass モードの **唯一の差分定義**。
各セルの「通常モード」列は workflow.md / 各 phase README からの引用。
bypass はこれらを **「待たずに即進む」** だけ。

| Phase | 通常モードの確認ステップ | bypass での扱い | 参照セクション |
|-------|--------------------|--------------|------|
| **Phase 1 (a)** | 必須情報（読者・目的・Before→After・分量・テーマ等）が不足していれば質問 | **必須 4 項目（読者・目的・分量・テーマ）が揃っていない時のみ 1 回だけまとめて質問**。それ以外（こだわり・トーン・参考素材等）はデフォルトで埋める。詳細は本ファイル「必須情報」セクション | phase1-hearing/README.md R1-1, R1-2 |
| **Phase 1 (b)** | 章立てメモ受領時：リファイン後の章割をユーザーに提示 → 承認を取る | リファイン後の章割を **1 度だけ提示**（即 Phase 2 に進む、承認待ちなし）。提示自体は **省略不可**（肥大化防止のため） | workflow.md「章立てメモを受け取ったら」R1-3 |
| **Phase 2** | Step 2-4：HTML 指示書を `decks/{slug}/plan.html` にファイル直書きしてユーザーに提示 → 「OK、進めて」を待つ | HTML 指示書は **通常通り decks/{slug}/plan.html にファイル直書きする**。M? VIOLATION 自動チェックも実施。提示と同時に Phase 3 へ進む（承認待ちなし） | workflow.md「Phase 2 — 情報設計」全節 |
| **Phase 3** | （ユーザー確認なし、自然遷移） | 通常モードと同一 | — |
| **Phase 4 (a)** | コンタクトシート目視 → 問題提示 → ユーザー判断（「OK」or「直して」） | Claude 自己目視 → 問題検出時は自動修正リトライ（最大 2 回、R-BYPASS-2）→ 解消しない / 3 件以上は進行停止して報告 | workflow.md「Phase 4 — ユーザー QA + decks/ 機械配置」+ qa/visual-qa.md + 後述「Phase 4 自動修正の詳細」 |
| **Phase 4 (b)** | 承認時に build-deck-package.js が機械実行 | Phase 4 自己 QA 成功時にそのまま `build-deck-package.js` を機械実行（通常モードと同一） | workflow.md「Phase 4 — ユーザー QA + decks/ 機械配置」R4-3 + phase4-qa/decks-packaging.md |

> 💡 **行番号でなくセクション名で参照する理由**: workflow.md は頻繁に編集されるため行番号は腐りやすい。セクション名なら見出し変更がない限りリンクが死なない。

**それ以外の処理（HTML 指示書の生成・テンプレ選定・参考情報集ページ・ファクトベース・
インライン番号・preview 生成・plan.html 同梱 等）はすべて通常モードと完全同一。**

## Phase 4 自動修正の詳細（bypass 固有のフォールバック）

通常モードが「Claude が問題提示 → ユーザー判断」なのに対し、bypass では Claude が判断を肩代わりする。

```
1. pptx-to-images.sh で PNG 群生成（通常モードと同一）
2. コンタクトシートで全体俯瞰 → 怪しいスライドのみ個別 view（通常モードと同一）
3. qa/visual-qa.md の基準で問題抽出（通常モードと同一）
4. 問題件数で分岐:
     - 0 件: そのまま `build-deck-package.js` を機械実行 → decks/ 配置
     - 1-2 件: 生成スクリプトを str_replace で自動修正 → 再ビルド → 再 QA（最大 2 回リトライ）
     - 3 件以上 or 自動修正で解消しない: 進行停止してユーザーに報告
5. リトライ上限到達時: 残存問題を記録して進行（完了レポートに警告として明記）し、decks/ 配置まで進める
```

**自動修正できる問題の種別**:
- テキストはみ出し（→ `shrinkText: true` 追加、padding 拡張、フォントサイズ微減）
- 余白不足（→ 座標の微調整）
- 軽微な重なり（→ y 位置の微調整）

**自動修正しない問題**（必ずユーザーに報告して停止）:
- 文言自体の過不足（削るべきか追加すべきかの判断は人間の仕事）
- カラーテーマのルール違反
- 指示書と実装の食い違い

## 完了時の報告フォーマット

```markdown
🎉 BYPASS MODE 完了

📦 成果物: decks/yyyy-mm-dd_{slug}/  (資料.pptx + plan.html + 生成メモ.md + preview/)

■ 実行サマリー（処理は通常モードと同一、確認待ちのみスキップ）
  Phase 1: 完了（章立てリファインを提示、承認待ちなしで通過）
  Phase 2: 完了（HTML 指示書を decks/{slug}/plan.html に書き出し、承認待ちなしで通過）
  Phase 3: 完了（N 枚生成）
  Phase 4: 自己 QA で X 件の問題を検出、Y 件を自動修正、Z 件が残存*
        + build-deck-package.js を機械実行 → decks/yyyy-mm-dd_{slug}/ に配置完了

*残存問題があれば、箇条書きでここに列挙（スライド番号・症状）

■ 中間ファイル
  <workdir>/
    ├── 01-design.html     ← Phase 2 で生成した HTML 指示書（plan.html と同一）
    ├── 02-deck.js         ← Phase 3 の生成スクリプト
    └── qa-images/         ← Phase 4 の PNG 群

何か違和感があれば、該当 Phase のログを元に修正できます。
```

## 安全弁まとめ

| # | 安全弁 | 目的 |
|---|-------|-----|
| 1 | 必須 4 項目が揃っていなければ 1 回だけ質問 | 情報不足で妙なデッキを作らない |
| 2 | Phase 4 で 3 件以上 or 自動修正で解消しない問題があれば進行停止 | 明らかに品質が悪い成果物を押し付けない |
| 3 | 全 Phase の中間成果物を `<workdir>/` に保存 | 失敗時の原因追跡・部分再実行を可能にする |
| 4 | 完了時に「bypass モードで走った」と明記＋残存問題を列挙 | 状況を隠さない（誤って配布されるのを防ぐ） |
| 5 | HTML 指示書は **通常モードと同じく必ず生成**（承認待ちのみスキップ） | 構成判断のトレーサビリティを残す |

## 使用例

### 例 1: 全情報を一度に渡す

```
/enostech-slides --bypass

FactHub v2.0 新機能リリース説明会の資料を作って。
- 読者: 既存ユーザーのデータアナリスト
- 目的: v2.0 の 3 つの新機能（自然言語検索拡張 / レポート自動生成 / 外部DB連携）の理解と、
        明日から使える実感
- 分量: 15 枚
- テーマ: enostech
- slug: facthub-v2-launch
```

→ 確認待ちなしで完走。最終的に `decks/yyyy-mm-dd_facthub-v2-launch/資料.pptx` のパスが提示される。
Phase 2 の HTML 指示書は途中で `decks/{slug}/plan.html` に書き出される（ユーザーはブラウザで開いて見るが、承認は問われない）。

### 例 2: 情報不足の場合

```
/enostech-slides --bypass

FactHub の資料作って。テーマは enostech、15 枚くらいで。
```

→ 読者と目的が不足 → 1 回だけまとめて質問 → 回答受け取り次第完走。

## バイパスモードを使うべきでないケース

以下のケースでは、bypass ではなく通常フローを推奨：

- **初見の分野・業種のデッキ**: 推定ミスのリスクが高い
- **投資家向け・対外プレゼンなど高ステークス**: Phase 2 / Phase 4 の人間確認が保険になる
- **過去に同様のデッキで修正が何度も発生した種類**: 自動修正だけでは品質不足になりがち

「何度も作っている社内ルーチン資料」「フォーマットが完全に固まっている定型物」 でこそ真価を発揮する機能。

---

## 複数デッキを並行で量産したいとき

スキル本体には並列実行ロジックを持たせていない。複数デッキを同時に進めたい場合は、
1 デッキ = 1 セッションの単位で、`--bypass` 付きの呼び出しを並列起動する：

- 別ターミナル / 別 Claude Code セッションで同時に投げる
- もしくは順次 1 件ずつ流す

スキル本体に subagent 並列を持つよりシンプルで、context duplicate のオーバー
ヘッドも避けられる。1 セッション内で N 件を順番に流す場合は、`--bypass` を
都度付けて呼び出すこと。
