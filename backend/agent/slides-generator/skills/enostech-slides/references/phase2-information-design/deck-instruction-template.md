# デッキ生成指示書テンプレート（情報設計リファレンス）

> 指示書は **2 タブ構成（最終アウトプット + レビュー）**。
> レビューは 4 サイクル × 各 1 ペルソナ + 専門レビュアー枠で実施 (詳細は `deck-instruction-schema.md` §M2)。
>
> - **見た目・構造の雛形** → `deck-instruction-html-template.html` を view で読んで踏襲
> - **JSON スキーマ仕様** → `deck-instruction-schema.md`
> - **中身の設計思想・項目定義** → このドキュメント（以下）

---

## 🔴 MUST ルール（絶対遵守・例外なし）

詳細は `deck-instruction-schema.md` 側に正本。要点だけ:

### M1. 本文は全文を書く（概要・メタ記述は絶対禁止）

各スライドの `detail_blocks` には、**実際にスライドに載る日本語一字一句**を完成原稿レベルで書く。「〜について説明」「〜を紹介」「〜に触れる」は絶対禁止。

### M2. レビューは 4 サイクル × 各サイクル 1 ペルソナ + 専門レビュアー枠

- `reviews[]` は 4 要素 (各サイクル 1 ペルソナ・別属性) + 専門レビュアー枠 (title-subcopy-qa)
- 詳細スキーマは `deck-instruction-schema.md` §M2

流れ: 1 人目でレビュー → 改善 → 別の 1 人で再読 → 改善 → ... → 専門レビュアーで仕上げ

### M3. 思考過程タブは作らない

`thinking` field は JSON に書かない。

### M4. 各スライドに挿絵判定 (`illustration_decision`) を書く

```json
"illustration_decision": { "adopt": true/false, "reason": "理由" }
```

### M5. 採用時は親スライドに `illustration` をネスト（独立 `S{id}-VISUAL` エントリは作らない）

`adopt: true` のスライドに、同じオブジェクト内へ `illustration: {scene, layout, ascii_art, intent, strength, refine_prompt}` をネストする。`S4-VISUAL` のような独立エントリを `slides[]` に作らない。付帯カード/ページはレンダ時・PPTX 生成時に自動 synthesize される。

### M6. 各スライドに `slide_goal: {title, subtitle}` を書く

「このスライドで何を伝えたいか」(title) と「読者をどう導きたいか」(subtitle) の 2 行。display の title/subtitle とは別の設計指針 field。

### 提出前の自己チェック

- [ ] すべての `detail_blocks.items` / `.text` が完成原稿（メタ記述ゼロ）
- [ ] `reviews[]` が 4 要素、各サイクルの `persona` が別属性
- [ ] `thinking` field が JSON に存在しない
- [ ] 全スライドに `illustration_decision` と `slide_goal`
- [ ] `adopt: true` の時だけ `illustration` がネスト、`adopt: false` には余計な `illustration` が残っていない
- [ ] `python3 scripts/render-deck-instruction.py` 実行時に stderr に MUST violation 警告が出ない

---

Phase 2（情報設計）の **最終アウトプット**。Claude はこのテンプレの情報を JSON に落として `decks/{slug}/plan.json` に保存し、`scripts/run-qa.py phase2 --plan decks/{slug}/plan.json` を経由して `decks/{slug}/plan.html` をユーザーに提示する。承認を得てから Phase 3 に進む。
すべての主張には **根拠（ソース URL or 社内情報の明記）** を添える。

## 使い方（Claude 向け指示）

### 生成フロー

1. Phase 1（ヒアリング）で得た情報 + 必要に応じて **web_search / web_fetch で補強取材** をして、以下のテンプレ全セクションを埋める
2. **`decks/{slug}/plan.json`** として書き出し、`run-qa.py phase2` で plan.html に変換
3. plan.html の URL をユーザーに提示し、「この指示書で Phase 3 に進んでよろしいですか? 修正点があればお聞かせください」と明示的に確認
4. ユーザーが承認（"OK"、"進めて"、"これで良い" など）するまで Phase 3 には **絶対に** 進まない
5. 修正依頼が来たら plan.json を update して `run-qa.py phase2` で plan.html を再生成、バージョンを v1 → v2 に上げる
6. 承認後、指示書を忠実に参照しながら Phase 3（PptxGenJS コード生成）へ

### ファクトの扱い

- **スライド内で言及する全ての主張** には根拠を添える:
  - 統計・数値データ → 調査元の URL + 調査年
  - ライブラリ・プロダクト名 → 公式サイト URL
  - 引用 → 元記事・発言者・日付
  - 市場データ → 信頼できる一次ソース
- **ソースが無い主張は「仮」と明記** し、ユーザーにソース追加を促す
- **ヒアリング中や Phase 2 の途中で**、事実確認が必要な主張が出てきたら、都度 web_search / web_fetch で裏取りする
- 社内情報・独自データの場合は `[社内: {出典}]` のように明記（例: `[社内: 2026年Q1売上実績]`）

### 指示書の品質チェック

plan.json を確定して run-qa.py phase2 を呼ぶ前に、Claude は以下を自己チェックする:

- [ ] Before → After が明確に書かれているか
- [ ] ストーリーラインで各スライドの流れが繋がっているか
- [ ] 各スライドに使うテンプレ ID が指定されているか
- [ ] 各スライドの主張に根拠（ソース or 社内情報）が紐づいているか
- [ ] 色トーン（gray/purple/amber）が感情設計と一致しているか
- [ ] 参照リンク一覧が末尾にまとめてあるか

## 2. ストーリーライン

デッキ全体の論理的な流れを 1 パラグラフで。「A だから B、B だから C…」の形で書く。

{例:
データ活用の課題は「量ではなく、実行者の不足」にある (1-4)。
→ 既存 BI ツールは「実行者を前提」としているから、根本解決にならない (5-7)。
→ FactHub は「実行者そのもの」を AI で再現するアプローチ (8-11)。
→ 実際の導入企業では、こう変わった (12-14)。
→ ゆえに、貴社の Q に対して、この形でお役に立てる (15)。}

### 流れのチェックポイント

- [ ] **冒頭 3 枚** で読者の問題意識を刺激できているか
- [ ] **中盤** で独自性と差別化を明示しているか
- [ ] **終盤** で次の行動 (CTA) に繋がっているか
- [ ] 各セクション間で「なぜ次の話になるのか」が明示されているか
- [ ] 読者の想定質問に対する答えが、適切な場所に置かれているか

## 3. スライド構成（ページごとの詳細設計）

各スライドは以下のフォーマットで記述する:

```
### S{番号} - {タイトル}
- **テンプレ**: {template_id (例: LIST-1 / VISUAL-3)}
- **トーン**: gray / purple / amber（サブコピー下敷きの色味）
- **主張**: このスライドが読者に伝える 1 メッセージ
- **根拠**: 主張の裏付け（[ソースリンク] or [社内情報])
- **接続**: 前スライドからの流れ + 次スライドへの橋渡し
- **タイトル**: （実際にスライドに載せる文字）
- **サブコピー**: （実際にスライドに載せる文字）
- **コンテンツ詳細**: テンプレに応じて具体要素を列挙
- **supplements** (挿絵スライド専用): 補強する先行スライドの id（例: `S7`）
- **挿絵** (挿絵スライド専用): scene_id / layout / ASCII アート / intent
```

> 💡 挿絵は **独立した 1 枚のスライド** として立てる（書籍の図解モデル）。
> 通常のコンテンツスライド内に挿絵ブロックを埋め込まない。詳しくは
> `references/_common/scene-patterns.md` の R-FIG-PRIORITY を参照。

### 挿絵スライドの記入例（書籍の図解モデル）

挿絵は **独立した 1 枚のスライド** として立てる。コンテンツスライドの中に埋め込まない。書籍の図版ページと同じ発想で、本文スライドの直後に挿絵スライドを差し込む:

```
### S7 - スキルは「暗黙知の結晶化装置」
- **テンプレ**: LIST-1 標準コンテンツ
- **トーン**: purple
- **主張**: SKILL.md に手順を落とすと属人化が解消され共有資産になる
- **根拠**: 社内ヒアリング（属人化により新メンバーの立ち上がりに平均 3 ヶ月）
- **タイトル**: 職人技を、形式知へ。スキルは"暗黙知の結晶化装置"
- **サブコピー**: 頭の中の「この場合はこう書く」を SKILL.md に落とすと、Claude が何度でもそれを再現してくれる。
- **コンテンツ詳細**:
  - 要点カード「スキル化 = 暗黙知 → 形式知 → 再現可能な資産」
  - 「スキル = あなたの"分身"を Claude に宿す」

### S8 - [挿絵] 結晶化のメタファー   ← S7 を補強する独立スライド
- **テンプレ**: VISUAL-3 ビジュアル主体（挿絵スライド専用）
- **supplements**: S7
- **タイトル**: （空 or 「結晶化」とだけ）
- **サブコピー**: （空）
- **コンテンツ詳細**: （空 or 最小キャプション 1 行）
- **挿絵**:
  ```
  [挿絵 | scene: before-after | layout: full-bleed | supplements: S7]
       Before           結晶化          After
  ┌─────────┐     ━━━→      ┌─────────┐
  │  職人    │               │SKILL.md │
  │  (斧)   │               │  ↓      │
  │ 1人の頭 │               │ チーム  │
  │ の中だけ │               │ で再現可│
  └─────────┘               └─────────┘
     属人化                    資産化
  intent: 結晶化のメタファーで SKILL.md の仕組みを直感的に
  ```

### S9 - 次のトピック ...
```

**ポイント**:
- S7 は本文スライド、S8 は **独立した挿絵スライド**。両方とも通し番号を持つ別スライド
- S8 の `template_id` は **`VISUAL-3`（ビジュアル主体）固定**
- S8 の `supplements: S7` で、この挿絵が何を補強するかを明示
- S8 の `detail_blocks` や `subtitle` は基本空。挿絵が単体で語る

### 挿絵スライドを立てる条件（迷ったら立てない）

1. 先行スライドが抽象的・概念的で、文字だけでは読者にピンと来ない
2. 視覚的メタファーで説明すると理解が進みそう
3. 先行スライドにダイアグラムがある場合、挿絵は**違う角度の表現**で補強する（重複禁止）
4. 全体の 10-30% 程度の枚数が目安。挿絵だらけにしない

図の選び方は `references/_common/scene-patterns.md` の R-FIG-PRIORITY を参照。

## 5. 参照リンク・出典一覧

本デッキで言及する全ての主張の出典を、整理して列挙する。Phase 3 でも参照できるよう、URL + タイトル + アクセス日のセットで記録。

> **ルール**: ここに列挙した出典は、デッキ末尾の **参考情報集ページ (DATA-4)** に `addReferenceTable` で反映される。
> スライド上では **URL は表示せず、タイトルが青文字ハイパーリンク** になる形式が標準。全デッキで必須の構成要素。
>
> **ルール**: さらに **本編スライドの本文内に `(1)` `(12)` のインライン参照番号（青文字リンク）** を主張の直後に埋め込む。
> 番号は下記 **「通し番号」列** の番号と一致させ、クリックで該当出典の URL に直接飛ぶ。
> これにより読者は「今見ているスライド → 出典」の経路を 1 クリックで辿れる。

### 通し番号の採番方針

- **SR ページに掲載する全出典に `(1)` から連番を振る**（カテゴリ横断でよい、重要度順でも出現順でもよい）
- 本編スライドの各主張には、対応する番号を `(1)` 形式で埋め込む
- **同じ出典を複数スライドで使う場合は同じ番号を使い回す**（番号は出典に付く、使用箇所に付くわけではない）
- Phase 3 で `inlineRef(num, url)` をテキスト run として `addText([...])` に混ぜて実装
